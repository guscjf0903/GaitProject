## 1. 백엔드 트러블슈팅

### 1.1 KAPT + Java 17+ IllegalAccessError

**증상**
- `./gradlew bootRun` 실행 시 `java.lang.IllegalAccessError` 발생
- `module jdk.compiler does not export com.sun.tools.javac.main...` 메시지

**원인**
- QueryDSL KAPT가 Java 17+ 모듈 시스템 하에서 `jdk.compiler` 내부 API에 접근하려 할 때 접근 제한에 걸림

**해결**

**파일**: `gradle.properties`

**이전 코드** (해당 설정 없음):
```properties
kotlin.code.style=official
```

**개선 후**:
```properties
kotlin.code.style=official

# KAPT + Java 17+ 호환: jdk.compiler 내부 접근 허용
org.gradle.jvmargs=-Xmx2g \
  --add-opens=jdk.compiler/com.sun.tools.javac.api=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.code=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.comp=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.file=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.main=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.parser=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.processing=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.tree=ALL-UNNAMED \
  --add-opens=jdk.compiler/com.sun.tools.javac.util=ALL-UNNAMED
```

### 1.2 timeline/at-commit 500 에러 (빈 리스트 JPA IN 절)

**증상**
- 새로고침 후 대화 내용이 전부 사라짐
- `GET /api/workspaces/{wid}/branches/{bid}/messages/timeline/at-commit` 호출 시 `500 INTERNAL_SERVER_ERROR`

**원인**
- `MessageService.timelineUpToCommit()`에서 조상 커밋 ID 리스트(`ids`)가 비어 있을 때
- Spring Data JPA의 `IN` 절에 빈 리스트를 전달하면 SQL 생성 실패 또는 예외 발생

**해결**

**파일**: `src/main/kotlin/com/gait/gaitproject/service/chat/MessageService.kt`

**이전 코드**:
```kotlin
val ids = ArrayList<UUID>(64)
var cur: Commit? = commit
while (cur != null) {
    ids.add(requireNotNull(cur.id))
    cur = cur.parent
}

val messages = messageRepository
    .findByWorkspace_IdAndCommit_IdInAndDeletedAtIsNull(workspace.id!!, ids)
    .sortedWith(compareBy({ it.createdAt }, { it.sequence }))
```

**개선 후**:
```kotlin
val ids = ArrayList<UUID>(64)
var cur: Commit? = commit
while (cur != null) {
    ids.add(requireNotNull(cur.id))
    cur = cur.parent
}

// JPA IN 절은 빈 리스트를 처리하지 못할 수 있으므로 early return
if (ids.isEmpty()) {
    return emptyList()
}

val messages = messageRepository
    .findByWorkspace_IdAndCommit_IdInAndDeletedAtIsNull(workspace.id!!, ids)
    .sortedWith(compareBy({ it.createdAt }, { it.sequence }))
```

### 1.3 커밋이 2~3번 몰아서 생성되는 현상

**증상**
- 커밋 버튼 클릭 또는 자동 커밋 시 동일한 내용의 커밋이 2~3개 연속 생성됨

**원인**
- USER/ASSISTANT 메시지 저장이 거의 동시에 발생할 때, 자동 커밋 조건(`pendingCount >= 20`)이 여러 요청에서 동시에 만족
- 경쟁 조건(race condition)으로 동일 브랜치에 대해 커밋 생성이 병렬 실행됨

**해결**

**1) BranchRepository에 락 메서드 추가**

**파일**: `src/main/kotlin/com/gait/gaitproject/domain/workspace/repository/BranchRepository.kt`

**이전 코드**:
```kotlin
interface BranchRepository : JpaRepository<Branch, UUID> {
    fun findByWorkspace_IdAndDeletedAtIsNullOrderByCreatedAtAsc(workspaceId: UUID): List<Branch>
    fun findByWorkspace_IdAndIsDefaultTrueAndDeletedAtIsNull(workspaceId: UUID): Branch?
    fun findByWorkspace_IdAndNameAndDeletedAtIsNull(workspaceId: UUID, name: String): Branch?
}
```

**개선 후**:
```kotlin
import jakarta.persistence.LockModeType
import org.springframework.data.jpa.repository.Lock
import org.springframework.data.jpa.repository.Query
import org.springframework.data.repository.query.Param

interface BranchRepository : JpaRepository<Branch, UUID> {
    // ... 기존 메서드 ...

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select b from Branch b where b.id = :id")
    fun findByIdForUpdate(@Param("id") id: UUID): Branch?
}
```

**2) CommitService에서 락 사용 + pending 없으면 예외**

**파일**: `src/main/kotlin/com/gait/gaitproject/service/workspace/CommitService.kt`

**이전 코드**:
```kotlin
val branch = branchRepository.findById(branchId).orElseThrow {
    NotFoundException("Branch not found. id=$branchId")
}
// ...
val pendingMessages = messageRepository.findByBranch_IdAndCommitIsNullAndDeletedAtIsNullOrderBySequenceAsc(branch.id!!)
// (pending 비어 있어도 커밋 시도)
```

**개선 후**:
```kotlin
val branch = branchRepository.findByIdForUpdate(branchId) ?: throw NotFoundException("Branch not found. id=$branchId")
if (branch.workspace.id != workspace.id) {
    throw IllegalArgumentException("Branch does not belong to workspace.")
}
// ...
val pendingMessages = messageRepository.findByBranch_IdAndCommitIsNullAndDeletedAtIsNullOrderBySequenceAsc(branch.id!!)
if (pendingMessages.isEmpty()) {
    throw IllegalArgumentException("No pending messages to commit.")
}
```

**3) MessageService 자동 커밋 시 예외 무시**

**파일**: `src/main/kotlin/com/gait/gaitproject/service/chat/MessageService.kt`

**이전 코드**:
```kotlin
if (pendingCount >= 20) {
    commitService.create(CommitCreateRequest(...), createdByUserId = user?.id)
}
```

**개선 후**:
```kotlin
if (pendingCount >= 20) {
    try {
        commitService.create(CommitCreateRequest(...), createdByUserId = user?.id)
    } catch (_: IllegalArgumentException) {
        // 동시 요청으로 이미 커밋이 처리된 경우(락 대기 후 pending=0 등) 무시
    }
}
```

### 1.4 워크스페이스 생성 시 초기 커밋 2개 생성

**증상**
- 새 프로젝트 생성 시 `initProject`와 `initial commit` 두 개의 커밋이 생성됨

**원인**
- 프론트엔드에 더미 커밋(`h6i7j`, `e4f5g`)이 하드코딩되어 있었고, 백엔드에서도 별도 초기 커밋 로직이 있어 중복 발생

**해결**

**파일**: `src/main/kotlin/com/gait/gaitproject/service/workspace/WorkspaceService.kt`

**개선 내용**:
- 워크스페이스 생성 시 `main` 브랜치에 **단일 `initProject` 커밋만** 생성
- `baseCommit`, `headCommit` 모두 이 커밋으로 설정

```kotlin
val initCommit = commitRepository.save(
    Commit(
        workspace = savedWorkspace,
        branch = main,
        parent = null,
        createdByUser = null,
        keyPoint = "initProject",
        shortSummary = null,
        longSummary = null,
        mergeType = MergeType.NONE,
        isMerge = false
    )
)
main.baseCommit = initCommit
main.headCommit = initCommit
branchRepository.save(main)
```

### 1.5 새 브랜치 생성 시 분기선이 보이지 않음

**증상**
- 새 브랜치를 만들어도 그래프에서 분기선이 표시되지 않음

**원인**
- `BranchService.create`에서 `baseCommitId`로 과거 커밋에서 브랜치를 따도, `headCommit`이 `baseCommit`과 동일하여 시각적 분기점이 없음

**해결**

**파일**: `src/main/kotlin/com/gait/gaitproject/service/workspace/BranchService.kt`

**이전 코드** (baseCommitId만 설정, headCommit = baseCommit):
```kotlin
baseCommit = selectedBaseCommit
headCommit = selectedBaseCommit  // 분기점 없음
```

**개선 후**:
```kotlin
baseCommit = selectedBaseCommit
headCommit = null  // 일단 null로 저장

// 저장 후 "Start {branchName}" 커밋을 별도 생성
if (selectedBaseCommit != null) {
    val startCommit = commitRepository.save(
        Commit(
            workspace = workspace,
            branch = saved,
            parent = selectedBaseCommit,
            keyPoint = "Start ${saved.name}",
            // ...
        )
    )
    saved.headCommit = startCommit
    branchRepository.save(saved)
}
```

## 2. 프론트엔드 트러블슈팅

### 2.1 브랜치 생성 후 새 브랜치가 화면에 안 보임 (새로고침 필요)

**증상**
- 새 브랜치를 만들고 `router.push()`로 이동해도 그래프/리스트에 새 브랜치가 표시되지 않음
- 새로고침(F5) 후에야 보임

**원인**
- `ChatPage.vue`가 같은 컴포넌트 인스턴스를 재사용하여 `onMounted()`가 다시 실행되지 않음
- `props.workspaceId`, `props.branchId` 변경 시 브랜치/커밋 목록을 서버에서 다시 불러오지 않음

**해결**

**파일**: `GaitProject_frontend/src/pages/ChatPage.vue`

**이전 코드**:
```javascript
onMounted(() => {
  // 서버에서 브랜치/커밋 로드
  ;(async () => {
    const branchRes = await BranchesService.listByWorkspace(props.workspaceId)
    // ...
  })()
  // ...
})
```

**개선 후**:
```javascript
const bootstrapFromServer = async () => {
  const seq = ++bootSeq
  try {
    const branchRes = await BranchesService.listByWorkspace(props.workspaceId)
    // ... 브랜치/커밋 로드 로직 ...
    commits.value = all
    await loadMessagesLatest(props.branchId)
    saveCurrentHeadSnapshot()
  } catch { /* ignore */ }
}

watch(
  () => [props.workspaceId, props.branchId],
  () => { void bootstrapFromServer() },
  { immediate: true },
)

onMounted(() => {
  // 테마, 키보드 이벤트만 처리
})
```

### 2.2 커밋 클릭 시 로딩 표시 없음 + 중복 클릭으로 커밋 2~3번 생성

**증상**
- 커밋 클릭 시 "대기 중" 표시가 없어 사용자가 연타
- 수동 커밋 모달에서 Enter/버튼 연타로 커밋이 여러 번 생성됨

**해결**

**1) 로딩 표시 추가**
```javascript
const loadingTimeline = ref(false)
const loadingCommitId = ref<string | null>(null)

const loadMessagesForCommit = async (commitId: string, branchId = props.branchId) => {
  loadingTimeline.value = true
  loadingCommitId.value = commitId
  try {
    const res = await MessagesService.timelineAtCommit(...)
    // ...
  } finally {
    loadingTimeline.value = false
    loadingCommitId.value = null
  }
}
```

**2) 커밋 버튼 중복 클릭 방지**
```javascript
const committing = ref(false)

const confirmCommit = () => {
  if (committing.value) return
  // ...
  ;(async () => {
    try {
      committing.value = true
      await createCommitOnServer(msg, { closeModal: true })
    } finally {
      committing.value = false
    }
  })()
}
```

```html
<button :disabled="committing" @click="confirmCommit">
  {{ committing ? 'Committing…' : 'Commit' }}
</button>
```

### 2.3 새 브랜치에서 대화 시 이전 브랜치 대화가 보임

**증상**
- 새 브랜치에서 대화를 이어가는데, 화면에는 분기 시점의 이전 브랜치 대화가 표시됨

**원인**
- 비동기 로딩 경쟁(race): 이전 브랜치의 `timeline` 응답이 늦게 도착해 새 브랜치 화면을 덮어씀
- 일부 커밋 객체에 `branchId`가 없어 HEAD 판별/checkout 로직이 꼬임

**해결**

**1) 타임라인 시퀀스 가드**
```javascript
let timelineSeq = 0

const loadMessagesForCommit = async (...) => {
  const seq = ++timelineSeq
  // ...
  const res = await MessagesService.timelineAtCommit(...)
  if (seq !== timelineSeq) return  // 늦게 온 응답 무시
  messages.value = ...
}
```

**2) 커밋 추가 시 branchId 항상 포함**
```javascript
appendCommitToGraph(newHash, parentHash, finalMsg, branchName, props.branchId)
// refreshCurrentBranchCommits에서도 branchId: props.branchId 추가
```

**3) 다른 브랜치 커밋 클릭 시 pendingCheckoutCommitId로 이동 후 로드**
```javascript
if (commit.branchId && commit.branchId !== props.branchId) {
  pendingCheckoutCommitId.value = commit.hash
  await router.push(`/w/${props.workspaceId}/b/${commit.branchId}`)
  return
}
// bootstrapFromServer에서 pendingCheckoutCommitId가 있으면 해당 커밋 로드
```

### 2.4 커밋 이동 후 최근 대화(미커밋)가 사라짐

**증상**
- 과거 커밋을 구경했다가 최신으로 돌아오면, 미커밋 5~6개가 안 보임
- 두 번째 클릭 시에야 보임

**원인**
- 커밋 이동 전 `saveCurrentHeadSnapshot()` 호출 누락
- `loadMessagesLatest`가 `timelineAtCommit` → `timelineAfter` 순차 호출로, 첫 렌더에 커밋만 보이고 pending이 늦게 붙음
- `serverHeadCommitId`가 커밋 배열 순서에 의존해 HEAD 판별이 틀어짐

**해결**

**1) checkout 전 스냅샷 저장**
```javascript
const checkout = async (commit) => {
  if (streaming.value) return
  saveCurrentHeadSnapshot()  // 항상 현재 화면 보존
  // ...
}
```

**2) loadMessagesLatest에서 커밋+미커밋 동시 요청**
```javascript
const [committedRes, pendingRes] = await Promise.all([
  headCommitId ? MessagesService.timelineAtCommit(...) : Promise.resolve({ data: [] }),
  MessagesService.timelineAfter(props.workspaceId, branchId, 0, 400),
])
if (seq !== timelineSeq) return

const committedList = (committedRes?.data ?? []).map(...)
const pendingMsgs = (pendingRes?.data ?? []).filter(m => !m.commitId).map(...)
messages.value = [...committedList, ...pendingMsgs]
```

**3) serverHeadCommitId로 HEAD 판별**
```javascript
const serverHeadCommitId = ref<string | null>(null)

// bootstrapFromServer, refreshCurrentBranchCommits, loadMessagesLatest에서 갱신
serverHeadCommitId.value = currentBranch?.headCommitId ?? null

// checkout 시
const headCommitId = serverHeadCommitId.value || commits.value.filter(...).slice(-1)[0]?.hash
if (headCommitId && commit.hash === headCommitId) {
  await loadMessagesLatest(props.branchId)
} else {
  await loadMessagesForCommit(commit.hash, props.branchId)
}
```

**4) 메시지 저장 실패 시 토스트**
```javascript
try {
  await MessagesService.send(workspaceId, branchId, { ... })
} catch (e) {
  toastNow('Message', e?.message ?? '메시지 저장 실패(서버). 새로고침/이동 시 사라질 수 있어요.', 2200)
}
```

**5) 로컬 스냅샷이 서버보다 길면 유지**
```javascript
if (headCommitId && localHeadSnap && localHeadSnap.length > messages.value.length) {
  messages.value = deepCopy(localHeadSnap)
  toastNow('Timeline', '서버에 아직 반영되지 않은 최근 대화가 있어 로컬 화면을 유지했어요.', 2200)
}
```

### 2.5 최신 커밋 클릭 시 첫 클릭에 미커밋이 안 보임 (두 번 눌러야 함)

**증상**
- 같은 브랜치에서 최신(HEAD) 커밋을 클릭해도, 첫 클릭에는 미커밋 내용이 안 보이고 두 번째 클릭에 보임

**원인**
- `loadMessagesLatest`가 `timelineAtCommit` → `timelineAfter` 순차 호출이라, 첫 렌더에 커밋만 표시되고 pending이 나중에 붙음
- `serverHeadCommitId`가 커밋 직후 갱신되지 않아 HEAD 판별이 잘못될 수 있음

**해결**

**1) createCommitOnServer 직후 serverHeadCommitId 갱신**
```javascript
appendCommitToGraph(newHash, parentHash, finalMsg, activeBranch.value, props.branchId)
serverHeadCommitId.value = newHash
```

**2) loadMessagesLatest에서 Promise.all로 동시 요청** (2.4와 동일)

## 요약 표

| 구분 | 증상 | 원인 | 해결 |
|------|------|------|------|
| 백엔드 | IllegalAccessError | KAPT + Java 17 모듈 접근 제한 | gradle.properties `--add-opens` 추가 |
| 백엔드 | timeline/at-commit 500 | JPA IN 절 빈 리스트 | `ids.isEmpty()` 시 early return |
| 백엔드 | 커밋 2~3번 생성 | 경쟁 조건 | BranchRepository `findByIdForUpdate`, pending 없으면 예외 |
| 백엔드 | 초기 커밋 2개 | 중복 로직 | WorkspaceService 단일 initProject |
| 백엔드 | 분기선 안 보임 | headCommit = baseCommit | BranchService "Start {name}" 커밋 생성 |
| 프론트 | 브랜치 생성 후 안 보임 | onMounted 재실행 안 됨 | watch(workspaceId, branchId) + bootstrapFromServer |
| 프론트 | 커밋 로딩/중복 | 표시 없음, 연타 | loadingTimeline, committing 가드 |
| 프론트 | 새 브랜치에 이전 대화 | race + branchId 누락 | timelineSeq, branchId, pendingCheckoutCommitId |
| 프론트 | 미커밋 사라짐 | 스냅샷/순차 로드/HEAD 판별 | saveCurrentHeadSnapshot, Promise.all, serverHeadCommitId |
