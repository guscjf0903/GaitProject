## 2026-03-01

### 한 일

- **데이터베이스 스키마 확장**
  - `Message` 엔티티에 `rawPrompt`, `rawResponse`, `inputTokens`, `outputTokens`, `totalTokens` 컬럼 추가.
  - Hibernate `ddl-auto=update`를 통해 DB 테이블 자동 반영.
  
- **백엔드 주도 메시지 저장 리팩토링 (Architecture Update)**
  - 기존: 프론트엔드가 사용자의 입력을 화면에 띄운 후 `MessagesService.send`를 호출하고, SSE가 끝나면 또 호출하여 저장하는 구조.
  - 변경: 프론트엔드는 렌더링과 스트리밍 요청(`POST /api/chat/stream`)만 담당. 
  - `ChatStreamController`가 요청을 받자마자 조립된 컨텍스트(`rawPrompt`)를 포함하여 USER 메시지를 직접 저장하고, 스트리밍 종료 후 토큰량과 함께 ASSISTANT 메시지를 직접 DB에 저장하도록 책임을 완전히 분리함.

- **Spring AI 토큰 메타데이터 로깅 연동**
  - `OpenAiChatAiService`에 `OpenAiChatOptions`를 명시적으로 주입하여 OpenAI API에 `streamUsage(true)`(또는 `stream_options: {"include_usage": true}`) 옵션을 활성화.
  - 스트리밍의 마지막 청크에 담겨오는 Usage 객체를 파싱하여 정확한 토큰 사용량을 산출.

- **프론트엔드 토큰 알림 UI 연동**
  - `sseChatStream.ts`에서 `ANSWER_DONE` 이벤트 발생 시 서버가 넘겨준 JSON 메타데이터를 파싱하도록 수정.
  - `ChatPage.vue`에서 스트리밍 종료 후 "이번 대화: OOO tokens 사용" 이라는 토스트 알림 노출.

### 결정/이유

- **왜 백엔드가 직접 저장하도록 변경했는가?**
  - 기존 방식은 백엔드에서 생성된 중요한 데이터(동적 조립된 `rawPrompt`, 실제 사용된 Token 수)를 DB에 넣기 위해, 프론트엔드로 데이터를 다시 내려보낸 뒤 프론트엔드가 이를 받아 API로 저장 요청을 해야 하는 매우 기형적이고 불안정한 구조였음.
  - 시스템 프롬프트 조립과 보안/과금 처리는 전적으로 백엔드의 책임이므로, 컨트롤러 레벨에서 즉시 저장하도록 리팩토링하여 데이터 정합성과 코드의 책임 소재를 명확히 함.

### 이슈/막힌 점

- **프론트엔드 OpenAPI 클라이언트 타입 불일치**: 백엔드 API 명세가 변경되어 `npm run generate:api`를 수행했으나, 자동 생성 과정에서 `list4` -> `list` 등으로 함수명이 변경되어 화면 렌더링 에러 발생. (수동으로 함수명 매핑하여 해결)
- **Spring AI의 EmptyUsage 이슈**: 기본적으로 Spring AI는 스트리밍 시 토큰 사용량을 내려주지 않아 `NULL`로 기록됨. `SPRING_AI_OPENAI_CHAT_OPTIONS_STREAM_USAGE=true` 설정 및 코드 레벨의 빌더 주입을 통해 해결.
- **환경변수 개행 오류로 인한 AI 실패**: `.env.local` 수정 중 줄바꿈이 누락되어 모델명이 `gpt-4o-miniSPRING...`으로 합쳐져 OpenAI API에서 `HTTP 400 (Invalid Model ID)` 에러 발생 및 연쇄적으로 Long Summary가 생성되지 않는 버그 발생. (환경변수 분리하여 해결)
- (상세한 트러블슈팅 과정은 `docs/troubleshooting/2026-03.md` 참조)
