## 2026-03-15 (2차) — RAG 파이프라인 전체 구현

### 개요

기존에 스텁(stub)으로만 존재하던 RAG(Retrieval Augmented Generation) 기능을 실제 동작하는 구조로 교체.
커밋 요약을 OpenAI 임베딩으로 벡터화하여 pgvector에 저장하고, 사용자가 과거 대화를 참조하는 질문을 하면
현재 브랜치 + 조상 커밋 범위 내에서 벡터 검색을 수행하여 관련 컨텍스트를 AI 프롬프트에 주입하는 구조.

### 한 일

#### 1. 임베딩 서비스 계층 (`service/ai/EmbeddingService.kt`) — 신규

- `EmbeddingService` 인터페이스 정의: `embed(text): List<Float>?`, `dimension(): Int`, `isAvailable(): Boolean`
- `DefaultEmbeddingService` 구현체:
  - Spring AI의 `EmbeddingModel` 빈을 `@Autowired(required = false)`로 주입. OpenAI API 키가 없는 환경에서도 서버가 정상 기동되도록 함.
  - `embed()` 내부에서 입력 텍스트를 8,000자로 truncate 후 `embeddingModel.embed(text)`를 호출하여 `FloatArray` → `List<Float>` 변환.
  - API 실패 시 `null` 반환(호출부에서 graceful 처리).
- 임베딩 모델: `text-embedding-3-small` (1536차원), `application.yml`의 `spring.ai.openai.embedding.options.model`로 설정 가능.

#### 2. 커밋 벡터 저장소 (`domain/workspace/repository/CommitVectorRepository.kt`) — 신규

- `NamedParameterJdbcTemplate` 기반의 네이티브 쿼리 전용 Repository.
- Hibernate가 pgvector 타입을 네이티브 지원하지 않으므로 JDBC 직접 사용.
- 주요 메서드:

| 메서드 | SQL 핵심 | 설명 |
|--------|---------|------|
| `saveEmbedding(commitId, embedding)` | `UPDATE commits SET embedding = CAST(:vec AS vector) WHERE id = :id` | 커밋 요약 임베딩 저장. `List<Float>`을 `[0.1,0.2,...]` 리터럴로 변환 |
| `findAncestorIds(headCommitId)` | `WITH RECURSIVE ancestors AS (...)` | HEAD에서 parent 체인을 재귀 CTE로 추적하여 모든 조상 커밋 ID 수집 |
| `countWithEmbedding(commitIds)` | `SELECT COUNT(*) ... WHERE embedding IS NOT NULL` | 스코프 내 임베딩 존재 여부 사전 확인(불필요한 API 호출 방지) |
| `searchSimilar(queryEmbedding, scopeCommitIds, limit)` | `embedding <=> CAST(:query AS vector) AS distance` | pgvector의 코사인 거리 연산자 `<=>`로 유사도 검색. `IN (:ids)` 절로 브랜치+조상 범위 제한 |

- 모든 메서드에 `try-catch` 래핑: pgvector 미설치 환경에서도 애플리케이션이 깨지지 않음.

#### 3. 커밋 생성 시 임베딩 자동 저장 (`service/workspace/CommitService.kt`) — 수정

- 생성자에 `EmbeddingService`, `CommitVectorRepository` 의존성 추가.
- `create()` 메서드 끝에 새 로직 추가:

```kotlin
// JPA 변경사항을 DB에 반영 후 JDBC 기반 임베딩 저장
commitRepository.flush()
generateAndSaveEmbedding(commit)
```

- `generateAndSaveEmbedding(commit)`:
  - `embeddingService.isAvailable()` false이면 즉시 return (OpenAI 키 없는 환경 안전).
  - 임베딩 대상 텍스트: `longSummary` → `shortSummary` → `keyPoint` 순으로 fallback.
  - `embeddingService.embed(text)` 호출 후 결과를 `commitVectorRepository.saveEmbedding()`으로 저장.
  - 임베딩 실패 시 warn 로그만 남기고 커밋 자체는 정상 완료됨.
- `commitRepository.flush()` 호출 이유: JPA의 write-behind 때문에 `commitRepository.save()` 시점에 실제 INSERT가 아직 DB에 반영되지 않았을 수 있음. JDBC `UPDATE`가 해당 row를 찾으려면 flush가 선행되어야 함.

#### 4. RAG 라우터 (`service/rag/RagRouter.kt`) — 신규

- `RagRouteType` enum: `DIRECT` / `FAST_RETRIEVAL` / `EXTRACTION_SEARCH` 3분기.
- 라우팅 로직:
  - `FREE` 플랜 → 무조건 `DIRECT` (RAG 미지원).
  - 22개 한국어/영어 키워드 매칭 (`"이전에"`, `"아까"`, `"찾아"`, `"기억"`, `"remember"`, `"search"` 등).
  - 매칭 0건 → `DIRECT`, 1건이면서 쿼리 길이 < 20자 → `FAST_RETRIEVAL`, 그 외 → `EXTRACTION_SEARCH`.
- `FAST_RETRIEVAL`은 기존 `ContextBuilder`의 Recent History/Lineage로 충분하므로 추가 검색 없이 통과.
- `EXTRACTION_SEARCH`일 때만 실제 벡터 검색 수행.

#### 5. RAG 검색 서비스 (`service/rag/RagService.kt`) — 신규

- `search(query, branchId, tokenBudget, limit)` → `RagSearchResult` 반환.
- 내부 흐름:
  1. `branchRepository.findById(branchId)` → `branch.headCommit.id` 획득.
  2. `commitVectorRepository.findAncestorIds(headCommitId)` → 조상 커밋 ID 목록.
  3. `commitVectorRepository.countWithEmbedding(ancestorIds)` → 0이면 검색 skip (API 호출 절약).
  4. `embeddingService.embed(query)` → 쿼리 벡터화.
  5. `commitVectorRepository.searchSimilar(queryEmbedding, ancestorIds, limit)` → Top-K 유사 커밋.
  6. `RagContextFormatter.format(results, tokenBudget)` → 프롬프트 삽입용 문자열.
- 각 단계에서 null/empty 시 `emptyResult()` 반환 (graceful 실패).
- `searchDurationMs` 측정하여 프론트에 전달.

### 결정/이유

- **JPA가 아닌 JDBC 사용 (CommitVectorRepository)**: Hibernate는 pgvector 타입(`vector(1536)`)을 네이티브 지원하지 않음. 기존 `Commit` 엔티티의 `embedding` 필드는 `insertable=false, updatable=false`로 선언되어 있어, 임베딩 저장/검색은 `NamedParameterJdbcTemplate`으로 네이티브 SQL을 직접 실행하는 것이 가장 안전하고 명확한 방법.
- **commitRepository.flush() 호출**: JPA의 write-behind 특성 때문에 `save()` 이후에도 실제 DB INSERT가 지연될 수 있음. JDBC `UPDATE`가 해당 row를 찾으려면 트랜잭션 내에서 JPA flush가 선행되어야 함.
- **Recursive CTE로 조상 탐색**: 기존 코드에서는 `Commit.parent`를 Lazy Loading으로 하나씩 추적했으나(N+1 쿼리), RAG에서는 조상 전체 ID가 필요하므로 PostgreSQL Recursive CTE를 사용하여 단일 쿼리로 해결.
- **임베딩 API 실패가 커밋 실패로 이어지지 않도록 설계**: `generateAndSaveEmbedding()`을 try-catch로 감싸서 임베딩 생성 실패(OpenAI API 다운, 키 미설정 등)가 커밋 저장에 영향을 주지 않도록 함.
- **빈 검색에 크레딧 미차감**: `ragResult.itemCount > 0`일 때만 `ragQuotaService.deduct()` 호출. 커밋 수가 적거나 임베딩이 없는 초기 상태에서는 사용자의 RAG 크레딧이 소비되지 않음.
- **3분기 라우팅 (DIRECT/FAST_RETRIEVAL/EXTRACTION_SEARCH)**: 모든 질문에 벡터 검색을 하면 API 비용과 지연이 증가하므로, 키워드 휴리스틱으로 사전 필터링. 대부분의 일상 대화는 `DIRECT`로 빠르게 통과하고, 과거 참조가 명시적인 질문만 검색을 수행.
